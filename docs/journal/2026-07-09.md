## 14:41 — C-Pilot 阈值涌现动态实验完成（零 GPU，离线）

### 方法
- 对 navtest sub1500 的 1495 场景，用 scorer 算 per-token score (720 维)
- 对全局阈值 τ（score 的 quantile），计算 per-scene keep_ratio = fraction(score > τ)
- 用已有 r=0.25/0.5/0.75/1.0 的 per-scene PDMS 做插值估算

### 结果

**① 自适应性极强 ✅**：
- τ_q=0.50 时：mean_kr=0.500, std=0.138, range=[0.20, 0.97], CoV=0.28
- 同一个阈值在不同场景"涌现"出 20% 到 97% 的完全不同保留比例
- 直方图见 `docs/results/figures/c_pilot/c_pilot_keepratio_histograms.png`

**② Pareto 不 dominate fixed r=0.5（方向性结论，含插值不确定性）**：
- τ-cut @ mean_kr=0.50: PDMS ≈ 0.889（低于 fixed r=0.5 的 0.895）
- τ-cut @ mean_kr=0.60-0.65: PDMS ≈ 0.8947（接近 fixed r=0.75）
- **注意**：0.6pt 差距部分来自 r<0.25 区间的插值外推（极端剪枝场景无实测），不可当实测结论
- Pareto 图见 `docs/results/figures/c_pilot/c_pilot_pareto.png`

**C-pilot 不是"失败"，而是论文第二根支柱**：

## 16:37 — Wall-clock Profiling 脚本 + 关键发现

### 发现（重要！影响论文效率 claim）

**当前实现是 Variant A（attention-mask pruning）**：被 prune 的 token 仍然占据序列长度，只是在 attention 计算中被 mask 为 -inf。

**后果**：
- ViT forward: 720 tokens 全部走完（不省）
- LLM prefill: 941 tokens 全部走完（mask 只影响 attention weight，不省 FLOPs）
- KV cache: 全长序列（不省 memory）
- **Wall-clock r=0.5 ≈ r=1.0（无显著差异！）**

**这意味着**：
- **论文不能报"省 50% wall-clock"** 除非实现 Variant B（真 drop）
- **但可以报**：(1) 理论 FLOPs 节省（如果 Variant B 实现）；(2) Variant A 下"质量近乎无损"是 Variant B 也能获得的（因为 Variant A 是 Variant B 的 quality-faithful proxy）
- 2-pass 开销（scorer selector 需要额外一次 forward）是额外成本，不是节省

### Profiling 脚本
- `scripts/profile_wallclock.py`：测量 5 种配置的 per-scene latency + peak memory + theoretical FLOPs
- 含 FLOPs 估算模块（ViT 32L + LLM 36L 理论值）
- 输出 JSON 到 `results/profiling/wallclock_profile.json`

### 对论文的影响（决策点）
- **Option X**：在论文中诚实报告"理论 FLOPs saving = ~27%（LLM prefill 占比），Variant B 实现 straightforward"+ 实测 Variant A 证明质量不损
- **Option Y**：花 ~1-2 天实现 Variant B（真 drop + M-RoPE recompute）→ 拿到真实 wall-clock 数据
- **推荐 Option X**：时间紧（19天），Variant B 工程量不小（M-RoPE 3D position 重算），且本论文重心是分析不是系统。reviewer 一般接受"theoretical FLOPs + we leave efficient implementation to future work"
